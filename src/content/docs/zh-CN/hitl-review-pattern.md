# 企业 Agent 的人工审核模式

## 为什么 HITL 重要

在企业与工业环境中，Agent 的输出可能影响运营、合规、安全或成本。Agent 应辅助决策，但不应在无人批准的情况下直接执行高风险操作。

HITL（Human-in-the-Loop，人在回路）不是安全的复选框，而是产品信任机制。每次被批准的推荐都在建立用户信心。每次被拒绝或修改的推荐都在改进系统。

**核心原则：** Agent 做数据密集型工作，人做运营决策。

## 操作风险等级

并非所有 Agent 输出都需要人工审核。风险分级决定什么进入 HITL：

| 风险等级 | Agent 输出示例 | 是否自动放行？ | 是否需要审核？ | SLA |
|---------|---------------|--------------|-------------|-----|
| **低** | 查询设备事件日志 | 是 | 否 | 不适用 |
| **中** | 生成根因假设 | 是（但记录日志） | 否（但记录日志） | 不适用 |
| **高** | 建议停产重新校准 | 否 | **是** | 30 分钟 |
| **关键** | 更改设备参数或紧急停机 | 否 | **是，双人审批** | 15 分钟 |

### 设计理由

- **低风险操作**（数据检索）应自动放行。对每次查询都要求审核会造成告警疲劳。
- **中风险操作**（分析、报告）自动放行但记录日志。如事后审查发现问题，日志可用于追溯审计。
- **高风险操作**（运营建议）必须路由到合格审核人。这些项目有 SLA 计时器 — 未审核项目会自动升级。
- **关键操作**（系统变更、安全决策）需要双人审批：两名独立审核人必须同时批准。

## 审核流程

```
Agent 推荐
  → 风险分级（低 / 中 / 高 / 关键）
    → 自动放行（低、中） → 记录日志并交付
    → HITL 队列（高、关键）
      → 人工决策：
        ├── 批准 → 执行并记录日志
        ├── 修改 → 应用更改、重新验证、执行并记录日志
        ├── 拒绝 → 记录原因、通知 Agent、重新调查
        └── SLA 超时 → 升级到上一级权限
  → 审计日志（每条决策均记录）
  → 最终报告（包含 HITL 决策轨迹）
```

## 审核人操作

审核界面应给审核人清晰的操作选项：

| 操作 | 何时使用 | 效果 |
|------|---------|------|
| **批准** | Agent 输出正确且完整 | 推荐原样执行 |
| **拒绝** | Agent 输出错误或不安全 | 返回 Agent 并附拒绝原因，要求重新调查 |
| **请求修改** | Agent 输出方向正确但需要改进 | Agent 修改特定部分而不重启完整调查 |
| **添加评论** | 审核人想补充领域背景知识 | 评论追加到报告而不阻塞执行 |
| **更改风险等级** | Agent 错误分类了风险 | 重新路由到正确队列并更新 SLA |
| **升级** | 审核人缺乏权限或背景信息 | 转移到上一级权限 |

### 审核人修改是功能，不是失败

在质量异常案例中，值班主管批准了核心建议（停产校准），但增加了一个补充措施："在重新校准期间检查模具对齐，依据 SOP-DIE-003 — 我们见过模具磨损导致该压机的漂移。"

这是 HITL 系统按设计正常工作。Agent 处理了数据关联，人工补充了 Agent 知识库尚未覆盖的领域知识。这是预期的协作模式 — 不是"人否决 Agent"，而是"Agent 做重活，人做判断。"

## 审核人角色

| 角色 | 范围 | 可批准 | 可推翻 |
|------|------|-------|-------|
| **值班主管** | 班次内的高风险操作 | 是 | Agent 推荐 |
| **运营经理** | 关键操作、已升级项目 | 是 | 主管决策 |
| **质量审核员** | 质量相关操作、合规 | 是 | 所有质量决策 |
| **工厂厂长** | 最终升级权限 | 是 | 所有决策 |

## 升级规则

如果分配审核人未在 SLA 内响应：

| 场景 | 主要审核人 | SLA | 升级目标 |
|------|----------|-----|---------|
| 标准高风险推荐 | 值班主管 | 30 分钟 | 运营经理 |
| 关键停产 | 运营经理 | 15 分钟 | 工厂厂长 |
| 低置信度假设（置信度 < 0.7） | 值班主管 | 30 分钟 | 运营经理 |
| SOP 与设备数据冲突 | 运营经理 + 质量审核员 | 30 分钟 | 工厂厂长 |
| Agent 超时 / 系统错误 | 值班主管 | 15 分钟 | IT 运维 |

### 升级原则

1. **有时限：** 每次审核都有 SLA。未审核项永远不会无限期阻塞系统。
2. **渐进式：** 升级沿权限链向上，而非横向转移。
3. **关键操作双人审批：** 安全关键操作需要两名独立审核人批准。
4. **优雅超时：** 如果所有升级层级都耗尽，结果以"未验证"标签交付 — 而非阻塞。
5. **审计追踪：** 每次升级均记录原因、时间戳和解决方案。

## 审计日志要求

每条 HITL 决策必须记录：

| 字段 | 描述 |
|------|------|
| 谁审核了 | 审核人 ID 和角色 |
| 推荐了什么 | Agent 的原始推荐 |
| 展示了什么证据 | 审核人可见的证据引用 |
| 做了什么决策 | 批准 / 修改 / 拒绝 |
| 何时发生 | 时间戳（ISO-8601） |
| 为什么 | 审核人陈述的理由 |
| SLA 状态 | 满足 / 违反 / 延期 |
| 最终报告版本 | 包含决策的报告链接 |

```json
{
  "event_id": "HITL-0053",
  "timestamp": "2024-03-18T15:04:00Z",
  "task_id": "TASK-0053",
  "agent": "quality_anomaly_agent",
  "risk_level": "high",
  "reviewer": "SUP-0088",
  "role": "shift_supervisor",
  "decision": "approved_with_modification",
  "rationale": "同意假设 1 和停产。补充：在重新校准期间依据 SOP-DIE-003 检查模具对齐。",
  "sla_minutes": 30,
  "response_time_minutes": 12,
  "sla_status": "met",
  "evidence_refs": ["EVT-010", "EVT-001", "CHUNK-003"]
}
```

## PM 设计笔记

### HITL 不只是安全层 — 它是产品信任机制

用户不信任第一天就自主行动的 Agent。HITL 创建了渐进的信任建立路径：Agent 推荐，人工批准，系统学习。随着时间推移，一贯正确的推荐可能为低风险类别赢得自动批准。

### 审核设计影响采用率

如果审核队列令人困惑、缓慢或嘈杂，审核人会绕过它。界面必须显示：Agent 想做什么、为什么（证据）、有多自信、有多紧急（SLA）、以及如果审核人不做任何事会发生什么（升级）。审核屏幕上的每条信息都是一个产品决策。

### 审核人应该看到证据，而不仅仅是结论

只显示"Agent 建议停产（置信度 0.92）"是不够的。审核人必须能够检查证据链 — Agent 使用的具体设备事件、测量值和 SOP 章节。这才能做出知情的判断，而非盲目的批准。

### 追踪决策质量随时间的变化

审核人决策是产品改进的数据源。如果某个 Agent 的推荐经常被修改或拒绝，这是改进提示、知识源或工作流设计的信号。HITL 数据关闭了 Agent 行为与产品质量之间的反馈回路。
