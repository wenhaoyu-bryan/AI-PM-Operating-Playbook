# 良率异常 RCA Agent 案例

## 案例概要

一条生产线检测到完全虚构的质量异常。Agent 使用模拟质量数据、设备事件、SOP 片段和工具上下文来调查该异常。它生成排序的根因假设，为每个假设引用证据，将风险建议路由到人工审核，并生成结构化的 RCA（根因分析）报告。

**所有数据均为合成数据。** 设备 ID、批次号、SOP 内容和事件日志均为虚构。本案例展示的是产品设计模式 — 而非真实的生产部署。

## 用户问题

产品设计需要解决以下用户视角的痛点：

| 角色 | 痛点 |
|------|------|
| **质量工程师** | 每次调查花费 2-4 小时在质量系统、设备日志和 SOP 文档之间手动关联数据 |
| **生产主管** | 必须在信息不完整的情况下批准高风险操作（如停产），往往缺乏充分证据 |
| **设备工程师** | 维护决策依赖隐性经验知识；不同设备间的重复性问题难以发现 |
| **AI 产品经理 / 平台负责人** | 需要在执行安全边界和建立用户信任的同时证明 Agent 的投资回报率 |

## 引入 Agent 之前：人工工作流

```
检测到质量异常
  → 工程师从质量系统拉取数据（15 分钟）
  → 工程师手动查询设备日志（30 分钟）
  → 工程师按关键词搜索 SOP 文档（20 分钟）
  → 工程师凭经验形成根因假设（时间因人而异）
  → 工程师写自由格式报告（30 分钟）
  → 如需停产：临时找主管审批（时间不定）
  → 无结构化审计追踪
  → 调查质量完全依赖个人经验
```

**核心问题：**
- 跨 3+ 系统的数据关联是手动且缓慢的
- 假设质量取决于工程师的资历
- SOP 搜索基于关键词，常返回不相关结果
- 高风险操作有时绕过正式审批
- 缺乏合规审查所需的结构化审计追踪

## Agent 辅助的工作流

Agent 不是替代工程师，而是自动化数据密集型的关联工作，将判断决策路由给人工：

| 步骤 | 执行者 | 描述 |
|------|-------|------|
| 1 | **传感器系统** | 检测到质量异常（自动传感器触发 Agent） |
| 2 | **规划 Agent** | 创建包含子任务和依赖关系的调查计划 |
| 3 | **质量异常 Agent** | 查询受影响批次的质量指标 |
| 4 | **质量异常 Agent** | 查询异常发生时间前后 ±4h 内的设备事件 |
| 5 | **SOP 助手** | 通过语义搜索检索相关 SOP 片段 |
| 6 | **质量异常 Agent** | 生成带有证据引用的排序根因假设 |
| 7 | **质量异常 Agent** | 按置信度评分对假设进行排序 |
| 8 | **审核 Agent** | 依据 7 维度评分标准对输出进行评分 |
| 9 | **HITL 系统** | 将高风险建议路由到人工审核队列 |
| 10 | **报告生成器** | 编制最终的结构化 RCA 报告 |
| 11 | **评估系统** | 对输出评分并将案例加入回归套件 |

## 合成案例示例

以下是一个完全虚构的示例，用于说明 Agent 的工作流：

| 字段 | 值 |
|------|-----|
| 异常 ID | ANO-007 |
| 批次 ID | B-2024-0053 |
| 产品 | PC-AL-200（精密铝外壳） |
| 产线 | LN-03 |
| 指标 | `厚度` |
| 测量值 | 2.52mm |
| 规格范围 | 2.00–2.30mm |
| 偏差 | 超出规格上限 +0.22mm |
| 严重程度 | **高** |
| 检测时间 | 2024-03-18 14:45 UTC |
| 检测方式 | 自动传感器 |

### Agent 检索到的证据

**设备事件（±4h 窗口内）：**

| 事件 ID | 设备 | 事件类型 | 严重程度 | 时间戳 | 描述 |
|----------|------|---------|---------|--------|------|
| EVT-010 | Press-03 | 校准漂移 | 高 | 14:30 | Z 轴偏移恶化至 +0.25mm |
| EVT-001 | Press-03 | 校准漂移 | 中 | 08:22 | 检测到偏移 +0.12mm（渐进模式） |
| EVT-004 | Press-03 | 压力下降 | 中 | 14:17 | 液压压力降至额定值的 85% |
| EVT-009 | Robot-04 | 定位误差 | 中 | 10:05 | 误差超出容差：0.08mm |

**SOP 匹配结果：**

| 片段 | SOP | 章节 | 相关度 | 关键内容 |
|------|-----|------|-------|---------|
| CHUNK-003 | SOP-PRESS-012 | 4.3（严重漂移） | 0.95 | 偏移 > 0.15mm → 停产，全面重新校准，主管签字 |
| CHUNK-002 | SOP-PRESS-012 | 4.2（轻微漂移） | 0.82 | 偏移 ≤ 0.15mm → 4h 内安排校准 |
| CHUNK-001 | SOP-PRESS-012 | 4.1（评估） | 0.71 | 记录偏移，对照容差表，确定严重程度 |

### 关键关联

1. **EVT-010 → ANO-007**：校准漂移（+0.25mm）在异常发生前 15 分钟检测到。测量偏差（+0.22mm）与偏移量减去正常工艺波动一致。
2. **EVT-001 → EVT-010**：同一设备在当天早上已出现漂移（+0.12mm）。6 小时内偏移从 +0.12mm 恶化至 +0.25mm — 明显的渐进性退化模式。
3. **历史关联**：ANO-001（同产线、同偏差类型、3 天前）曾出现 +0.12mm 偏移。这是**重复性问题** — 而非孤立事件。

### 排序假设

Agent 生成 3 个假设并给出置信度评分：

| 排名 | 假设 | 置信度 | 风险等级 | 需要 HITL |
|------|------|-------|---------|----------|
| 1 | Press-03 Z 轴渐进性校准漂移导致厚度偏差 | 0.92 | 高 | **是** |
| 2 | 液压压力下降加剧了校准漂移效应 | 0.45 | 中 | 否 |
| 3 | Robot-04 定位误差 — 相关性弱，不同设备单元 | 0.15 | 低 | 否 |

**假设 1** 正确识别了根因：偏移（+0.25mm）超过 SOP-PRESS-012 第 4.3 节的阈值（0.15mm），需要停产和全面重新校准。Agent 正确地将假设 3 降级（Robot-04 是不同的设备单元，无因果联系）。

## Agent 输出结构

最终 RCA 报告采用标准化的 10 节结构：

1. **执行摘要** — 发生了什么，根因，建议，审核人决策
2. **异常详情** — 测量数据，规格限值，检测方式，严重程度
3. **调查时间线** — 触发、工具调用、假设、审核的日志
4. **设备关联** — 相关设备事件的时间分析
5. **SOP 引用** — 带有相关度评分的相关程序章节
6. **根因假设** — 按置信度排序，附带证据数组
7. **建议措施** — 具体步骤，附 SOP 引用、风险等级和审批状态
8. **证据附录** — 所有引用事件、测量和 SOP 片段的完整数据
9. **HITL 决策** — 审核人身份，决策（批准/修改/拒绝），理由，响应时间
10. **审计日志** — 完整工具调用追踪及时间戳

## 产品设计决策

本节记录了设计中嵌入的 PM 判断：

### 为什么 Agent 不应该直接停止产线

停产会带来运营和财务影响。Agent 可以*建议*停产并引用要求停产的 SOP，但决策必须来自了解更广泛生产背景（下游订单、人员安排、安全）的人。Agent 的角色是呈现证据；人的角色是做出运营决策。

### 为什么需要证据引用

没有引用，Agent 的输出与幻觉无法区分。每个假设必须引用具体数据点（设备事件、测量值、SOP 章节）。这使得输出可审计 — 工程师可以通过检查引用源来验证每个声明。

### 为什么显示置信度

带有"92% 置信度"的答案比不加限定的断言更诚实。置信度评分向审核人传达应该给予 Agent 输出多少权重。它们还创建了自然的升级路径：低置信度输出不论风险等级都路由到人工审核。

### 为什么风险建议需要 HITL

这不是安全的复选框 — 而是产品信任机制。如果用户第一次看到 Agent 建议停产，且完全没有人工审核，用户就再也不会信任这个系统了。HITL 是逐步建立信任的：每次被批准的推荐都成为 Agent 可靠的证据。

### 为什么评估是工作流的一部分，而非事后补救

Agent 质量是可衡量的，不是凭感觉的。每次调查输出都根据 7 维度评分标准评分。暴露弱点的案例成为回归测试用例。没有这一步，修复一个场景的提示变更可能悄无声息地破坏另一个场景。

### 为什么 MVP 使用模拟数据而非真实集成

产品模式必须先行验证，然后再做任何集成工作。模拟数据使团队能够在等待真实生产系统 API 访问之前测试工作流、HITL 路由和报告结构。这降低了产品投资的风险：如果工作流在干净的合成数据下都不合理，在混乱的真实数据下也行不通。

## 引入前后对比

| 指标 | 人工流程 | 有 Agent | 改进幅度 |
|------|---------|---------|---------|
| 形成假设所需时间 | 2–4 小时 | < 5 分钟 | **约 96% 更快** |
| 咨询的证据来源 | 2–3 个（因人而异） | 5+ 个（系统性） | **更全面** |
| 时间关联分析 | 手动，常被遗漏 | 自动化 ±4h 扫描 | **不错过模式** |
| SOP 覆盖 | 关键词搜索，有遗漏 | 语义搜索，排序输出 | **相关内容可被呈现** |
| 假设格式 | 自由格式，依赖经验 | 多假设带置信度 | **结构化、可比较** |
| HITL 执行 | 随意 | 自动基于风险的路由 | **100% 合规** |
| 审计追踪 | 手动笔记 | 自动工具调用日志 | **完全可追溯** |
| 报告一致性 | 自由格式，作者不同风格不同 | 10 节结构化模板 | **标准化** |

## 本案例展示了什么

本案例为面试场景设计。它展示了：

- **场景思维：** 将混乱的真实调查转化为清晰的 Agent 工作流
- **架构素养：** 多 Agent 设计与工具注册表和知识源
- **安全判断力：** 知道在哪里划定自主边界
- **评估严谨性：** 用结构化评分标准衡量 Agent 质量，而非凭感觉
- **产品务实性：** 为 MVP 选择模拟数据，在规模化前建立回归套件
