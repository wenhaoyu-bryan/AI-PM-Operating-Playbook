# 智能体产品设计

> 设计目标驱动的 AI 工作流——包含工具、知识、边界、人工审核和可衡量的结果。

---

## 智能体不是聊天机器人

智能体与聊天机器人的区别在于结构，而非仅仅在于能力。

聊天机器人是无状态的逐轮交互。用户提一个问题，模型生成一个回答。每一轮对话相互独立，没有持久目标、没有工作流、也没有工具调用。价值体现在单次回复本身。

智能体产品则根本不同。它将以下要素组合为一个完整的系统：

- **目标** — 智能体在多个步骤中追求的明确结果
- **领域知识** — 关于业务领域、用户和当前情境的信息
- **决策序列** — 不是一次调用，而是结构化的推进过程
- **工具调用** — 读取、写入、搜索、计算或调用外部系统的能力
- **状态** — 在步骤间保留的信息，而非每轮重置
- **边界** — 明确定义智能体可以做什么、不可以做什么
- **反馈** — 来自环境或检查机制的信号，用于指导下一步行动
- **人工介入** — 由人类审核、批准或纠正的检查点

聊天机器人回答问题。智能体完成工作。产品设计的核心挑战是：定义智能体应该做什么、不应该做什么，以及人如何始终掌控结果。

## 何时适合使用智能体

并非所有问题都需要智能体。是否使用应该经过审慎判断，基于工作本身的性质。

### 适合使用智能体的场景：

- **工作需要多个步骤。** 单次查询不需要智能体。需要收集信息、推理分析、采取行动并验证结果的工作流才需要。
- **路径无法完全预定义。** 如果每个步骤都事先已知，确定性工作流或脚本更简单、更可靠。当序列取决于中间结果时，智能体才有价值。
- **需要访问工具或系统。** 智能体需要读取数据库、调用 API、搜索知识库或与外部服务交互来完成推理。
- **中间证据很重要。** 最终输出的质量取决于沿途收集的信息和做出的决策的质量。智能体必须能够评估证据，而不仅仅是生成回复。
- **工作流受益于自适应推理。** 同样的触发可能因上下文不同而走向不同路径，智能体应根据发现的信息调整方法。

### 不适合使用智能体的场景：

- **确定性规则即可解决。** 如果输入到输出的映射是固定的且定义明确，规则引擎或简单脚本更快、更便宜、更可预测。
- **简单搜索或表单即可满足。** 如果用户只需查找文档或提交表单，专用界面比智能体循环更清晰、更不容易出错。
- **错误代价高且无审核环节。** 如果错误操作无法轻易纠正——金融交易、法律文件、安全关键系统——智能体不应自主执行。
- **操作不可观测或不可逆。** 如果你无法查看智能体做了什么或撤销操作，风险就太高。每个智能体操作都应可追溯，且尽可能可逆。

正确的问题不是"智能体能不能做这件事？"，而是"这项工作是否能从智能体的结构——目标驱动、多步推理、工具使用、自适应行为——中获益，足以证明额外的复杂性是值得的？"

## 智能体产品画布

智能体产品画布是定义智能体产品做什么以及如何运作的设计工具。它包含十个维度，每个维度都应在实施开始前得到具体回答。

### 1. 目标

智能体追求什么结果？这是最重要的设计决策。目标必须足够具体以指导智能体行为，足够可衡量以评估成效。"帮助用户"或"提供洞察"这类模糊目标无法产生可靠的智能体。

### 2. 触发条件

什么启动工作流？触发条件可以是用户请求、定时事件、来自其他系统的信号，或某个阈值被触发。触发条件决定了智能体何时启动以及接收到什么初始上下文。

### 3. 用户 / 操作者

谁在工作流中与智能体交互，谁在事后监督它？这两者可能是不同的人。用户提供建议和接收结果。操作者监控绩效、审核决策，并随时间调整系统。

### 4. 上下文与知识

智能体完成工作需要哪些信息？包括领域知识、用户历史、系统当前状态、相关文档和任何先前决策。上下文使智能体的行为具备情境适切性，而非泛化通用。

### 5. 工具

智能体可以访问哪些系统？每个工具代表一种能力：读取数据库、写入电子表格、调用 API、执行计算、搜索知识库。工具集应当经过刻意设计——智能体可以调用的每个工具都是一种能力，每个缺失的工具都是一种边界。

### 6. 工作流与状态

智能体做出的决策序列是什么？它在步骤间保留哪些信息？工作流定义了智能体行为的结构。状态使智能体能够在先前步骤的基础上继续，而非每次都从头开始。

### 7. 自主性边界

智能体在哪里独立行动？在哪里需要审批？这是最重要的安全设计决策。高置信度、低影响的操作可以自主执行。高影响、不可逆或不确定的操作应要求人工审批。

### 8. 人工审核

哪些检查点需要人工判断？人工审核不是自动化的失败——它是一种设计选择。好的智能体产品让人能够方便地检查中间结果、批准或拒绝拟议操作、并提供纠正以改善未来行为。

### 9. 评估

如何衡量成功？评估应超越回答质量本身。衡量工作流完成率、决策准确率、工具使用模式、人工覆盖频率和端到端的结果质量。指标应反映智能体是否达成了目标，而非仅仅是否产生了看似合理的回复。

### 10. 失败与恢复

出错时会发生什么？每个智能体都会遇到无法处理的情况——数据缺失、指令模糊、工具故障、意外输入。产品必须定义智能体如何检测失败、失败时做什么（重试、上报、停止），以及系统如何恢复。

## 参考工作流

以下工作流描述了智能体产品的一般结构。并非每个智能体都遵循每一步，但该模式足够通用，可作为起点。

```
信号或用户请求
    |
    v
收集上下文
    |
    v
制定计划
    |
    v
调用工具
    |
    v
评估证据
    |
    v
提出行动建议
    |
    v
需要时进行人工审批
    |
    v
执行、记录、学习
```

**信号或用户请求。** 工作流从触发条件开始——用户消息、外部系统事件或定时检查。触发条件提供初始意图。

**收集上下文。** 智能体收集完成工作所需的信息：相关文档、系统当前状态、用户历史、领域知识。这一步决定了后续所有步骤的质量。

**制定计划。** 基于目标和可用上下文，智能体确定要采取的步骤。计划可以是显式的（书面列表）或隐式的（结构化推理链）。重要的是智能体在行动之前已有策略。

**调用工具。** 智能体执行计划中的步骤——查询数据库、读取文档、执行计算、调用 API。每次工具调用都是现实世界中的操作，应给予相应的审慎对待。

**评估证据。** 收集信息后，智能体评估证据是否充分、结果是否一致、计划是否需要调整。这是自适应推理步骤——也是智能体区别于简单脚本的关键所在。

**提出行动建议。** 智能体推荐具体的下一步：草拟回复、建议编辑、推荐决策。建议应包含背后的推理和支持的证据。

**需要时进行人工审批。** 根据产品画布中定义的自主性边界，部分操作在执行前需要人工审核。审核界面的设计应与产品的其他部分一样用心。

**执行、记录、学习。** 审批通过后（或自主操作在建议后），智能体执行操作、记录做了什么及原因，并存储对未来工作流实例可能有用的信息。

## 产品反模式

以下是智能体产品设计中的常见模式，应当避免。

**将单次 LLM 响应称为"智能体"。** 单次提示-响应对不是智能体，而是聊天机器人调用。智能体有目标、工作流、工具、状态和评估。将聊天机器人重新包装为智能体不会改变产品——只会改变预期。

**从模型能力出发而非从用户决策出发。** "我们有一个模型能做 X，所以围绕 X 构建产品"是能力驱动设计。它产生的产品在技术上令人印象深刻，但没有解决具体问题。应从用户需要做出的决策出发，再确定需要什么能力。

**赋予工具却不定义权限。** 智能体能访问数据库意味着可以读取和写入。如果你不定义它能读哪些表、能写哪些表、绝不能碰哪些表，那你给智能体的不是工具——而是风险。

**添加记忆却不定义该记住什么。** 记忆并不总是有益的。如果智能体记住一切，它会积累噪音。如果什么都不记，它就无法从先前交互中学习。应定义什么该记、记多久、以及在什么条件下该遗忘。

**自动化高影响操作却不设审核。** 当操作可逆、影响小且定义明确时，自动化是有价值的。在没有审核检查点的情况下自动化高影响操作，不是功能——是隐患。

**只评估回答质量而非工作流结果。** 在错误工作流中的正确回答是糟糕的产品。应评估智能体是否达成了目标，而非仅仅是单次回复是否合理。衡量完成率、决策准确率、用户对结果的满意度，以及人工纠正的频率。

**向用户隐藏不确定性和证据。** 当用户能看到推理过程时，他们会更信任智能体。隐藏不确定性——以与充分支持的结论相同的置信度呈现不确定的结论——会侵蚀信任，并使用户更难提供有用的纠正。

**在确定性自动化更优的场景中构建智能体。** 如果工作流是固定的、输入是结构化的、输出是可预测的，脚本或规则引擎更简单、更快、更便宜、更可靠。智能体仅在工作流需要自适应推理、非结构化输入或动态工具使用时才能增加价值。

## 与 AI PM Canvas 的关联

智能体产品画布与本手册其他部分使用的 12 维度 AI PM Canvas 直接对应。这种关系是结构性的——智能体产品画布是 AI PM Canvas 针对智能体类产品的专门化。

| 智能体产品画布 | AI PM Canvas |
|---|---|
| 目标 | Decision to Support（决策支撑） |
| 上下文与知识 | Data / Knowledge Sources（数据/知识来源） + Object Model（对象模型） |
| 工具 | AI / Agent Capability（AI/智能体能力） |
| 工作流与状态 | Workflow Boundary（工作流边界） |
| 自主性边界 | Workflow Boundary（工作流边界） + Failure Modes（失败模式） |
| 人工审核 | Human Review Points（人工审核点） |
| 评估 | Evaluation Criteria（评估标准） |
| 失败与恢复 | Failure Modes（失败模式） |
| 触发条件 | Business Scenario（业务场景） |
| 用户 / 操作者 | User / Operator（用户/操作者） |

如果你已经为产品填写了 AI PM Canvas，智能体产品画布是针对智能体类产品的细化和扩展。如果还没有，先从 AI PM Canvas 开始——它提供了更广泛的框架，智能体画布在此基础上构建。
