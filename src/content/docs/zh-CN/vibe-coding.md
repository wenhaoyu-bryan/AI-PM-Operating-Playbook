# Vibe Coding：面向产品探索与原型的 AI 辅助开发

**[English](../en/vibe-coding.md)** | **中文**

用编码智能体把产品意图转化为可测试的原型——但不要把原型和生产软件混为一谈。

> **在本手册中**，Vibe Coding 指 AI 辅助的产品原型开发——PM 定义产品意图、范围、约束、示例和验收标准，编码智能体协助实现。这不意味着不加理解或验证地接受生成的代码。

> 产品判断 + 编码智能体 = 更快的原型与学习循环

---

## 什么是 Vibe Coding？

**定义**：产品经理定义产品意图、架构边界和设计约束，编码智能体负责代码生成、调试和迭代。PM 始终对"构建什么"和"为什么构建"负责。

**核心价值**：
- AI PM 可以独立构建许多早期原型，减少产品发现阶段对工程的依赖
- 产出的不是代码，而是对问题域的更深理解
- 通过构建来理解系统，而不是仅靠阅读规格说明

**适用场景**：
- 交互原型与工作流演示
- 技术探索与架构可行性验证
- 风险可控的内部工具
- 产品概念验证
- 面向工程交接的可执行规格说明

## 实践案例

### 一个本体产品实验

**背景**：作为 AI PM 进入半导体行业，需要通过动手实践来理解本体驱动的企业平台的底层逻辑。

**做法**：
- 一个人、一个编码智能体（Claude）、两周的迭代循环
- 没有 Sprint Planning，没有 Stand-up——从产品意图直接到原型的工作流

**产出**：
- 一个展示图谱实体关系的本体原型
- 一套完整的架构认知框架
- 一篇记录领域洞察的技术文章

**关键洞察**：
> "构建本体系统最难的部分不是代码，是心智模型。理解图数据库不是本体，本体不是 Schema，Schema 不是数据模型——这些是同心圆的抽象层级，搞对它们是产品问题，不是工程问题。"

### 一个工业智能体平台原型

**背景**：在投入工程资源之前，验证制造业数字员工协同平台的可行性。

**做法**：
- PM 定义的工作流草图通过编码智能体转化为可交互的原型
- 通过可运行的代码而非静态图表进行架构探索
- 基于利益相关者对原型本身的反馈进行迭代优化

**产出**：
- 一个可导航的原型，作为具体的沟通载体
- 更精确的需求，减少了后续工程冲刺中的歧义
- 更清晰的平台边界决策——该做什么、不该做什么

## Vibe Coding 的关键技能

### 1. 架构思维

**核心能力**：
- 定义系统边界
- 指定层间契约
- 推理数据流和失败模式

**实践方法**：
- 在向智能体发出指令之前先画架构图
- 定义清晰的 API 接口和数据契约
- 识别原型中哪些部分需要忠实于生产约束，哪些可以简化

### 2. 质量判断

**核心能力**：
- 评估智能体输出的正确性和完整性
- 识别潜在问题、缺失的边界条件和架构风险
- 判断原型何时"足以用来学习"与何时"足以用来发布"

**实践方法**：
- 在每次迭代前建立质量检查清单
- 测试关键交互路径和边界条件
- 验证架构决策在真实数据下是否站得住

### 3. Prompt 工程（辅助技能）

**核心能力**：
- 将产品意图转化为编码智能体可执行的结构化指令
- 设计清晰的约束、验收标准和示例
- 通过显式规格说明而非侥幸心理来处理边界情况

**实践方法**：
- 用结构化、分层的方式描述需求（意图、范围、约束、示例）
- 提供具体的示例和反例
- 提前明确技术约束和非功能性期望

> Prompt 工程是一项辅助技能——核心价值来自产品判断、领域理解和架构清晰度。好的 Prompt 是清晰思考的结果，而非其替代品。

## Vibe Coding 的工作流

### 阶段 1：需求定义

```markdown
## 产品意图
[一段话描述产品定位以及这个原型需要回答的问题]

## 架构边界
- 输入：[什么数据进来]
- 处理：[怎么处理]
- 输出：[什么结果出去]

## 设计约束
- 技术栈：[例如 React + Vite + Tailwind CSS]
- 性能要求：[例如 响应时间 < 100ms]
- 范围限制：[这个原型刻意不覆盖什么]
```

### 阶段 2：架构设计

```markdown
## 系统架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端      │ →   │   后端      │ →   │   数据库    │
│  React      │     │  FastAPI    │     │  SQLite     │
└─────────────┘     └─────────────┘     └─────────────┘
```

## 数据流
1. 用户输入 → 前端组件
2. 前端组件 → API 调用
3. API 调用 → 后端处理
4. 后端处理 → 数据库查询
5. 数据库查询 → 结果返回

## API 契约
- GET /api/items → 返回数据列表
- POST /api/items → 创建新数据
- PUT /api/items/:id → 更新数据
- DELETE /api/items/:id → 删除数据
```

### 阶段 3：代码生成

```markdown
## Prompt 模板

你是一个资深前端工程师，擅长用 React + Vite + Tailwind CSS 快速构建原型。

基于以下架构设计，生成一个可运行的项目。

要求：
1. 使用 React 19 + Vite + Tailwind CSS 4
2. 所有组件都要是 functional component + hooks
3. 样式全部用 Tailwind CSS
4. 数据用 mock 数据
5. 为 API 调用添加基本的错误处理

架构设计：
[粘贴阶段 2 的架构设计]
```

### 阶段 4：迭代优化

```markdown
## 迭代 Prompt

基于当前代码，进行以下修改：

1. [修改点 1]：[具体描述]
2. [修改点 2]：[具体描述]

约束条件：
- 保持现有架构不变
- 只修改指定部分
- 确保其他功能不受影响
- 标注你所做的假设
```

## 最佳实践

### 1. 先理解再构建

**错误做法**：
- 没有心智模型就让智能体直接写代码
- 不理解架构就开始改
- 遇到问题就从头开始

**正确做法**：
- 先理解业务需求以及原型需要回答的问题
- 先设计或草拟系统架构
- 先定义接口契约和数据模型

### 2. 定义清晰的边界

**错误做法**：
- 模糊的需求描述
- 不明确的技术约束
- 没有验收标准

**正确做法**：
- 结构化的需求文档，包含明确的范围限制
- 明确的技术约束和非功能性期望
- 具体的验收标准，用来判断原型何时"完成"

### 3. 迭代而非重写

**错误做法**：
- 遇到问题就从头重写
- 不保留上一次迭代中有用的代码
- 重新生成一切导致架构决策丢失

**正确做法**：
- 在现有基础上迭代
- 保留和重构有用的代码
- 逐步优化和改进，记录变更内容和原因

### 4. 验证而非信任

**错误做法**：
- 完全信任智能体的输出而不加检查
- 不测试就使用生成的代码
- 不验证边界条件和错误处理

**正确做法**：
- 测试关键交互路径
- 验证边界条件
- 检查错误处理和失败模式
- 检查生成的代码中是否有明显的安全或逻辑问题

## 杠杆效应的来源

PM 主导的原型开发，其价值不在于原始速度，而在于：

- **减少翻译损耗**——理解问题的人可以直接指导实现，减少意图到原型之间的信息损失
- **更早验证交互和工作流假设**——在工程团队锁定架构之前
- **让需求可执行、可检查**——一个可运行的原型能消除文档无法消除的歧义
- **在工程承诺之前允许多次低成本迭代**——每次循环都会使需求更精确
- **通过可运行的演示改善沟通**——比抽象的规格说明更有效

> 实际收益因范围、技术复杂度、数据可用性和生产要求而异。有些原型可以在几小时内完成，有些需要数天的仔细迭代。关键在于更快的学习，而非固定的时间保证。

## 注意事项与边界

### 适合

- 交互原型与工作流演示
- 技术探索与可行性验证
- 风险可控、范围明确的内部工具
- 与利益相关者进行产品概念验证
- 可执行的规格说明，用于工程交接

### 需要工程审查

这一类工作可以由 PM 主导启动原型，但在任何真实环境部署之前必须经过工程审查：

- 涉及真实用户数据的任何内容
- 认证和授权逻辑
- 外部 API 集成
- 持久化数据库和数据迁移
- 业务关键工作流
- 超出受控内部演示的部署

### 不适合仅作为原型工作

这些领域从一开始就需要工程主导。PM 原型可以用于沟通，但绝不能被误认为是可用的系统：

- 金融交易或货币计算
- 受监管的决策（医疗、法律、合规）
- 安全关键的工业控制系统
- 敏感个人数据处理
- 生产安全架构
- 无人监督的自主高影响动作

## 总结

**核心定位**：
- AI 原生的 PRD 同样可以作为编码智能体的指令集
- 原型可以作为可交互的需求规格说明
- 对话式的迭代可以缩短某些原型迭代循环

**PM 主导原型开发带来的变化**：
- AI PM 可以独立构建许多早期原型
- 这减少了产品发现阶段对工程的依赖，而非生产阶段对工程的需求
- 工程师在架构、安全、可靠性、可维护性、性能、集成和运维方面仍然不可或缺
- PM 主导原型开发的价值在于更快的学习和更精确的沟通——而非取代工程专业能力

**工具链**：
- 需求输入：Markdown
- 原型生成：编码智能体（Claude 等）
- 典型技术栈：React + Vite + Tailwind CSS（或任何智能体可以脚手架搭建的技术栈）
