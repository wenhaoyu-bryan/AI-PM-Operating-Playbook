# 工具、知识与工作流模式

## 核心理念

工业 Agent 需要对工具和知识的结构化访问。产品设计应定义 Agent 可以读什么、可以生成什么、可以建议什么，以及 — 最重要的是 — 不能执行什么。

这是 Agent 设计的操作层。它位于"用户想要什么"（场景）和"Agent 如何行为"（提示）之间，决定了 Agent 的实际能力和安全边界。

**核心原则：** 工具设计就是产品设计。权限设计就是产品设计。Agent 能做什么、不能做什么是 PM 的决策，而非工程的马后炮。

## 工具注册表

工具注册表定义了每个 Agent 可用的能力，包括风险等级、输入/输出模式和是否需要人工批准。

### 工具注册表示例

| 工具 ID | 描述 | 输入 | 输出 | 风险等级 | 需要 HITL |
|---------|------|------|------|---------|----------|
| `query_quality_anomalies` | 按批次 ID 和时间范围查询质量偏差记录 | `batch_id`, `time_range` | 附带测量值和规格限的异常记录 | 低 | 否 |
| `query_equipment_events` | 按设备 ID 和时间窗口查询设备事件日志 | `equipment_id`, `start_time`, `end_time` | 附带时间戳和描述的事件记录 | 低 | 否 |
| `search_sop` | 对 SOP 知识库进行语义搜索 | `query`, `max_results` | 附带相关度评分的排序 SOP 片段 | 低 | 否 |
| `get_equipment_context` | 检索设备元数据和维护历史 | `equipment_id` | 设备档案、最近维护日期、常见故障模式 | 低 | 否 |
| `generate_report` | 将 Agent 发现编制为结构化报告 | `template`, `findings` | 包含所有章节的格式化报告 | 中 | 否（但记录日志） |
| `dispatch_alert` | 向运营团队发送告警 | `severity`, `message`, `recipients` | 附带投递状态的告警确认 | **高** | **是** |

### 工具设计原则

1. **最小权限：** Agent 只能访问其需要的工具。SOP 助手拿不到 `dispatch_alert`。
2. **显式风险：** 每个工具有声明的风险等级。这自动驱动 HITL 路由。
3. **模式定义：** 输入输出模式使工具行为可测试、可文档化。
4. **可审计：** 每次工具调用均记录输入、输出、时间戳和调用 Agent。

## 知识源

知识源决定了 Agent"知道"什么以及能引用什么作为证据。知识源的质量和结构直接影响 Agent 的可信度。

| 来源 | 格式 | 访问模式 | 示例内容 |
|------|------|---------|---------|
| **设备事件** | CSV / 时序 | 按设备 ID 和时间范围查询 | Press-03 在 14:30 的校准漂移（+0.25mm 偏移） |
| **质量异常记录** | CSV / 结构化 | 按批次 ID 和偏差类型查询 | 批次 B-2024-0053：厚度 2.52mm，规格 2.00–2.30mm |
| **SOP 文档** | 文本片段 | 按上下文语义搜索 | SOP-PRESS-012 第 4.3 节：严重漂移校正程序 |
| **历史报告** | 结构化文档 | 按设备、产线或偏差类型查询 | 用于重复性问题检测的历史 RCA 报告 |
| **领域本体** | JSON 图 | 术语查找和关系遍历 | 设备层级、故障模式分类 |

### 知识源设计原则

- **结构很重要：** 带有章节编号的、良好分块的 SOP 比原始文本转储产生更好的引用。
- **新鲜度很重要：** 过时的知识源产生错误的推荐。每个知识源都需要 `last_updated` 字段。
- **空白必须声明：** 如果知识源不覆盖某个场景，Agent 应该说不知道 — 不要猜。
- **质量影响信任：** 如果用户前三次检查证据引用都发现错误，他们再也不会检查了。

## 工作流设计

产品经理定义工作流 — 有序的 Agent 任务序列、依赖关系、决策闸门和故障路径。

### 工作流模板

对每个 Agent 工作流，定义：

| 元素 | 描述 | 示例 |
|------|------|------|
| **触发条件** | 什么启动工作流 | 自动传感器检测到质量偏差 |
| **Agent 任务** | 每个 Agent 做什么，按什么顺序 | 数据采集（并行）→ 假设生成 → 审核 → 报告 |
| **依赖关系** | 哪些任务等待哪些其他任务 | 假设生成等待数据采集 + SOP 检索 |
| **工具访问** | 每个任务可以使用哪些工具 | 质量异常 Agent：`query_equipment_events`、`search_sop` |
| **证据要求** | 输出中必须引用什么 | 每个假设引用 ≥ 1 个设备事件 + ≥ 1 个 SOP 章节 |
| **输出格式** | 最终输出的结构 | 10 节 RCA 报告 |
| **审核边界** | 哪些输出路由到 HITL | 高风险建议；置信度 < 0.7 |
| **评估标准** | 如何衡量输出质量 | 7 维度加权评分标准 |

### 质量异常 RCA 工作流

```
触发：自动传感器检测到质量偏差
  │
  ├── T1：查询质量指标 [并行，低风险，无 HITL]
  ├── T2：查询设备事件（±4h 窗口） [并行，低风险，无 HITL]
  ├── T3：检索 SOP 片段 [并行，低风险，无 HITL]
  │
  └── T4：生成根因假设 [依赖 T1+T2+T3，中风险，无 HITL]
        │
        └── T5：审核员评分 [中风险，无 HITL]
              │
              ├── 通过 → T6：生成最终报告
              └── 高风险 / 低置信度 → HITL 队列 → T6（附带审核人决策）
```

**设计说明：** T1、T2 和 T3 并行运行，因为它们彼此没有依赖关系。T4 等待所有三个任务，因为假设必须引用设备数据、质量指标和 SOP 上下文。这种并行性比顺序流水线减少大约一半的延迟。

## Agent 通信协议

Agent 通过结构化的任务对象进行通信 — 而非自由形式的对话：

```json
{
  "task_id": "TASK-0053",
  "agent": "quality_anomaly_agent",
  "status": "completed",
  "input": {
    "anomaly_id": "ANO-007",
    "batch_id": "B-2024-0053"
  },
  "output": {
    "hypotheses": [...],
    "risk_level": "high",
    "requires_hitl": true
  },
  "tool_calls": [
    { "tool": "query_equipment_events", "timestamp": "...", "result_count": 4 },
    { "tool": "search_sop", "timestamp": "...", "result_count": 3 }
  ],
  "evidence": [
    { "type": "equipment_event", "ref": "EVT-010", "relevance": "..." },
    { "type": "sop_section", "ref": "CHUNK-003", "relevance": "..." }
  ]
}
```

**为什么结构化任务对象重要：** 它们支持依赖追踪、重试逻辑、审计日志和 Agent 间交接。它们还使系统可观察 — 产品经理可以检查任何任务并确切了解发生了什么。

## 故障模式设计

Agent 工作流必须优雅地处理故障。每个故障模式都需要定义的恢复路径：

| 故障模式 | 检测方式 | 恢复路径 | 用户影响 |
|---------|---------|---------|---------|
| 设备数据缺失 | `query_equipment_events` 返回空集 | 扩展时间窗口：±4h → ±8h → ±24h。仍为空则标记为"上下文有限" | 报告附带数据空白声明 |
| 未匹配到 SOP | `search_sop` 返回零个高于阈值的结果 | 扩大搜索词范围。返回最接近匹配并附免责声明 | 报告附带"无精确 SOP 匹配"说明 |
| 假设冲突 | 两个假设置信度差距 < 0.15 | 呈现两者，提升风险等级，强制 HITL 路由 | 附带证据的并排对比 |
| Agent 置信度低于阈值 | 主要假设置信度 < 0.7 | 以扩展数据重试。仍然低则路由到 HITL | 最佳可用假设附带不确定性警告 |
| Agent 超时或崩溃 | 在预期窗口内无输出 | 最多重试 2 次。然后升级到 HITL | 部分结果通知 |

### 故障设计原则

1. **宁可报错，也不静默失败：** 每次故障都以清晰的解释呈现给用户。
2. **优雅降级：** 部分结果优于零结果。
3. **人工升级：** 不确定性和冲突总是路由到 HITL。
4. **审计追踪：** 每次故障和恢复操作均记录。
5. **不编造数据：** 当数据缺失时，Agent 应如实说明 — 绝不凭空填补空白。

## PM 设计笔记

### 工具设计就是产品设计

工具注册表不是工程实现细节。它是定义 Agent 能做什么、不能做什么以及在什么条件下需要人工批准的产品产物。一个无法读懂工具注册表的 PM 无法推理 Agent 的安全边界。

### 权限设计就是产品设计

哪个 Agent 可以访问哪个工具，在什么条件下，以什么风险等级 — 这些都是产品决策。它们反映了关于什么安全、什么有价值以及自主边界应划定在哪里的产品判断。

### 知识源质量影响 Agent 信任

如果 SOP 知识库过时或不完整，Agent 将做出错误推荐 — 不是因为 AI 不好，而是因为产品输入不好。知识源管理是 PM 的职责，而不仅仅是数据工程的任务。

### Agent 应暴露中间推理产物

用户应能看到：调查计划、工具调用、收集的证据、考虑的假设、审核状态和最终报告。这些是产品 UI 产物，而非隐藏的思维链。它们通过使 Agent 的推理可检查来建立信任。
