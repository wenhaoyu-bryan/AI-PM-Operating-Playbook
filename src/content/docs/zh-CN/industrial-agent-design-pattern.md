# 工业 Agent 产品设计模式

## 为什么需要这个模块

大多数 AI Agent 演示止步于对话交互。用户提问，Agent 回答。这对于客服机器人没问题，但在工业和企业环境中就行不通了。

工业 Agent 产品需要结构化的工作流、证据链、工具边界、人在回路（HITL）审核闸门、安全分级、权限控制、可审计性和评估 — 这些都不是聊天机器人架构能提供的。

本模块是从一个公开的 Industrial Agent Product Sandbox 项目改编而来的脱敏案例模式。它展示了 AI 产品经理如何将工业知识工作（质量异常调查、SOP 检索、设备事件关联）转化为结构化的 Agent 产品系统。

**核心信息：** 工业 Agent 不是聊天机器人。它们是结合了业务事件、Agent 规划、工具注册表、知识检索、证据链、人在回路审核、安全边界和评估的工作流系统。

## 核心产品模式

```
业务事件
  → Agent 规划器
    → 工具注册表
    → 知识检索
    → 证据链
    → 根因假设
    → HITL 审核
    → 最终报告
    → 评估与审计日志
```

每一步都是产品设计决策，而非工程实现细节。AI PM 需要定义：

- 什么触发 Agent（业务事件）
- Agent 可以访问什么（工具注册表 + 知识源）
- 必须引用什么证据（证据链要求）
- 谁审核什么（HITL 边界）
- 如何衡量质量（评估）
- 记录什么（审计追踪）

## 工业 Agent 与聊天机器人的本质区别

| 维度 | 聊天机器人 Agent | 工业 Agent |
|------|----------------|-----------|
| **环境** | 低风险问答 | 高风险运营 |
| **输出** | 流畅回答 | 结构化证据 + 假设 |
| **工具访问** | 无限制或单一 API | 有风险等级的作用域工具注册表 |
| **安全性** | 内容过滤 | 风险分级 + HITL 路由 |
| **证据** | 可选，常不引用 | 必须引用，每个结论都需数据支撑 |
| **人工审核** | 无 | 内置于工作流中，管控高风险操作 |
| **审计追踪** | 不需要 | 每次工具调用、决策和审核均记录 |
| **评估** | "回答是否有帮助？" | 7 维度加权评分标准，有通过/失败阈值 |
| **交互方式** | 对话驱动 | 工作流驱动 |

工业 Agent 运行在错误建议可能导致停产、安全事故或产品缺陷的环境中。产品架构必须反映这一现实 — 不是让 Agent 更强大，而是围绕它设计正确的边界。

## 核心产品对象

这些对象构成了 AI PM、工程师和领域专家在设计工业 Agent 系统时的共同语言：

| 对象 | 是什么 | 为什么重要 | 示例（质量异常案例） |
|------|-------|----------|-------------------|
| **Agent** | 具有特定工具、知识和输出的限定 AI 角色 | 定义边界；防止一个"万能 Agent"包揽一切 | 质量异常 Agent、SOP 助手、审核 Agent |
| **工具** | 具有输入/输出模式、风险等级和权限的注册能力 | 使 Agent 能力显式化、可审计 | `查询设备事件`、`搜索SOP`、`生成报告` |
| **知识源** | Agent 可访问的结构化数据或文档 | 决定 Agent"知道"什么、能引用什么 | 设备事件 CSV、SOP 片段、质量异常记录 |
| **工作流** | 具有依赖关系和决策闸门的有序 Agent 任务序列 | 将人工流程转化为 Agent 可执行的步骤 | 触发 → 数据采集 → 假设 → 审核 → 报告 |
| **任务** | 分配给特定 Agent 的工作单元，有定义的输入输出 | 支持并行执行、依赖追踪和重试逻辑 | "查询批次 B-2024-0053 的 ±4h 设备事件" |
| **证据** | 支持或反驳假设的引用数据点 | 使 Agent 推理可检查；审计和合规所需 | EVT-010 在异常前 15 分钟的校准漂移事件 |
| **审核** | 在风险操作执行前对 Agent 输出的人工评估 | 安全闸门；防止高风险决策自动执行 | 值班主管批准/拒绝/修改停产建议 |
| **报告** | 包含执行摘要、证据、假设和建议的结构化输出 | 为运营、合规和质量团队标准化输出格式 | 10 节 RCA 调查报告 |
| **评估案例** | 具有预期行为的测试场景，在变更前把控质量 | 防止回归；捕获提示漂移；强制执行质量标准 | EVAL-002：高风险升级与 HITL 审核 |
| **审计日志** | 每次 Agent 操作、工具调用、审核决策和时间戳的不可变记录 | 合规、事件分析和流程改进所需 | 完整追踪：规划器 → 工具调用 → 假设 → HITL 决策 → 报告 |

## 这对 AI PM 展示了什么

本案例模块展示了以下 AI PM 能力：

| 能力 | 展示内容 |
|------|---------|
| **场景抽象** | 将真实工业工作流（质量调查、SOP 检索）转化为结构化 Agent 场景，不复制专有流程 |
| **Agent 架构设计** | 定义 Agent 角色、工具注册表、知识源和协作模式 — 不仅仅是提示词工程 |
| **企业工作流设计** | 设计具有并行执行、依赖图、重试逻辑和故障模式的多步骤工作流 |
| **HITL 设计** | 风险分级（低/中/高/关键）、升级 SLA、审核人角色和审计要求 |
| **安全与权限思维** | 工具级访问控制、提示注入防御、数据分级和最小权限原则 |
| **评估设计** | 加权多维度评分标准、结构化评估案例、校准流程和回归测试节奏 |
| **跨职能沟通** | 在领域专家（质量工程师）、工程师（Agent 架构）和利益相关者（运营经理）之间进行翻译 |
| **产品判断力** | 知道 Agent 不应该做什么 — 在哪里划定自主边界，何时升级给人工 |
