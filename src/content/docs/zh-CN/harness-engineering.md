# Harness Engineering：面向 AI 辅助产品交付的约束体系设计

**[English](harness-engineering.md)** | **中文**

> 设计上下文、约束、工具、检查和反馈机制，让编码智能体可靠地工作。

---

## 什么是 Harness Engineering？

Harness Engineering（面向编码智能体的上下文、工具、约束与反馈体系设计）是一门设计编码智能体**运行环境**的工程学科。大语言模型能够生成和推理；智能体在模型之上增加了循环——规划、行动、观察和迭代；Harness 则为智能体循环提供上下文、工具、约束、检查机制、记忆和恢复机制，使其能够产出可靠的结果。

真正的提升不来自一个更长的提示词。它来自建立一个分阶段的工作流，其中每个阶段产出的工件约束并改善下一个阶段。

### Model、Agent 和 Harness

| 层级 | 职责 |
|------|------|
| **Model（模型）** | 生成文本、推理代码、起草方案 |
| **Agent 循环** | 规划步骤、执行动作（编辑文件、运行测试）、观察结果、迭代改进 |
| **Harness（约束体系）** | 提供仓库指令、产品上下文、架构决策、工具访问、范围约束、自动化检查和会话记忆 |

Harness 是智能体周围的**环境**——不只是提示词。它包括项目文件、CI 检查、工具定义、决策日志以及引导工作从想法到交付的流程结构。

## Harness 的六个组成部分

一个设计良好的 Harness 由六个部分组成：

### 1. 上下文（Context）

仓库指令、产品意图、架构决策、领域术语，以及具体的正例和反例。上下文是智能体开始工作之前读取的内容。

上下文工件示例：
- `CLAUDE.md` 或 `AGENTS.md` ——编码智能体的顶层指令
- `docs/product-context.md` ——用户是谁、解决什么问题、什么算成功
- `docs/architecture.md` ——系统边界、技术栈、关键模式

### 2. 工作流（Workflow）

分阶段的工作序列：头脑风暴、规格说明、实施规划、增量执行、审查和修正。每个阶段产出一份书面工件，作为下一阶段的输入。

### 3. 工具（Tools）

智能体可用的能力：文件访问、终端命令、测试运行器、代码检查工具、浏览器或截图工具、文档查询。工具集应有意识地选择——智能体能调用的每个工具都是一种能力，缺失的每个工具都是一条边界。

### 4. 约束（Constraints）

范围限制、架构边界、安全限制、非目标和不变量。约束防止智能体扩大范围、引入不支持的模式或违反安全要求。当约束以项目级指令中的显式规则形式存在时最为有效。

### 5. 感知与评估（Sensors and Evaluation）

为智能体提供反馈的自动化检查：构建检查、测试套件、代码检查、视觉审查、验收标准和回归检测。感知机制关闭反馈回路——没有它们，智能体无法知道自己的输出是否正确。

### 6. 状态与交接（State and Handoff）

项目记忆、决策日志、实施计划、进度工件、版本控制历史，以及为下一次会话准备的上下文。状态是让工作能够跨会话、跨人员持续的关键。

## 分阶段工作流

```
模糊的想法
    |
    v
结构化头脑风暴
    |
    v
达成一致的产品方向
    |
    v
书面规格说明
    |
    v
实施计划
    |
    v
增量编码智能体执行
    |
    v
构建 / 测试 / 视觉审查
    |
    v
内容与产品判断
    |
    v
更新 Harness 并重复
```

每个阶段产出具体工件：头脑风暴摘要、产品方向文档、规格说明、实施计划、可运行代码、测试结果、审查记录。下一阶段继承上一阶段的约束和上下文。这就是让智能体工作可靠的原因——不是更好的提示词，而是更好的结构。

## AI PM 在 Harness 中的角色

Harness Engineering 不纯粹是工程学科。产品经理拥有与意图、范围和判断相关的特定部分。

**AI PM 负责的部分：**

- 产品意图和用户/业务上下文
- 范围定义和非目标
- 交互和工作流需求
- 正例和反例
- 验收标准和评估指标
- 人工审查点——在哪些节点需要人工检查后才能继续
- 业务和产品风险评估
- 利益相关者叙事——何时沟通什么

**仍需工程判断的部分：**

- 安全架构和威胁建模
- 基础设施和部署设计
- 可扩展性和性能特征
- 生产数据处理和隐私保护
- 可观测性、日志和告警
- 代码可维护性和技术债务
- 集成可靠性和故障恢复
- 事件响应流程

PM 定义什么是好的。工程定义如何让它安全可靠。

## 项目结构示例

Harness 存在于代码仓库本身。具体结构因项目而异，但模式是一致的：

```
CLAUDE.md / AGENTS.md          # 顶层智能体指令
docs/
  product-context.md            # 用户是谁，解决什么问题
  architecture.md               # 系统边界、技术栈、设计模式
  decisions.md                  # 关键决策及理由记录
  acceptance-criteria.md        # 本项目的完成定义
  plans/                        # 每个工作阶段的实施计划
tests/                          # 智能体可运行的自动化检查
scripts/                        # 辅助脚本（构建、部署、检查）
```

**每个工件的用途：**

- **`CLAUDE.md` / `AGENTS.md`** ——智能体首先读取的内容。包含项目级规则、约束和约定。
- **`product-context.md`** ——让智能体在跨会话中保持对用户需求和业务目标的一致理解。
- **`architecture.md`** ——记录系统设计决策，防止智能体引入不兼容的模式。
- **`decisions.md`** ——持续记录决策理由，防止智能体重新讨论已确定的选择。
- **`acceptance-criteria.md`** ——定义完成标准。智能体和人类都使用此文件来评估输出。
- **`plans/`** ——逐步的实施计划，将工作拆分为可审查的增量。
- **`tests/`** ——为智能体提供代码是否正常工作的反馈。
- **`scripts/`** ——自动化工具，保持反馈循环快速和一致。

## 核心原则

### 结构优于提示词

一个提示词无论多么详尽，都无法替代结构化的工作流。先搭建脚手架——智能体会使用它。

### 工件优于对话

每个重要决策都应写在智能体可以读取的文件中，而不是埋在聊天记录里。文档会留存。对话不会。

### 约束优于指令

告诉智能体不要做什么，往往比告诉它做什么更有效。显式约束——范围限制、禁止模式、非目标——缩减了可能的错误空间。

### 反馈优于信任

永远不要假设智能体的输出是正确的。建立检查机制——测试、代码检查、视觉审查、验收标准——让正确性可验证而非靠假设。

### 增量优于整体

将工作拆分为小的、可审查的步骤。每一步都应产出可测试的结果。这让错误尽早可见，修正成本更低。
